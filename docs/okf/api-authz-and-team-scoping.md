## The invariant

`backend/src/api/rest/router.ts` is one 1706-line file: every route, and all authn/authz, live inline — there is no middleware directory. Three caller models are recognised purely from the request shape:

- **Session Bearer.** `resolveUserId` (router.ts:915-923) reads `Authorization: Bearer <token>`, looks it up in `prisma.session`, and throws if missing or expired. `requireAuth` (router.ts:959-966) wraps it: returns the userId, or sends 401 and returns `null` so the handler can early-return. (`resolveStreamUserId`/`requireStreamAuth`, router.ts:930-952, is a header-or-`?token=` variant used only by the SSE `/stream/logs` route, because `EventSource` cannot set headers; the URL token is redacted from the access log by the http logger in `app.ts`.)
- **X-API-Key.** `authenticateApiKey` (router.ts:145-170) reads `X-API-Key`, resolves `prisma.logSource.findUnique({ where: { apiKey } })`, and 403s if the resolved source's `id` does not match the URL's `:sourceId` — the middleware's own comment names this explicitly: "any valid key could forge logs into another source / team (cross-tenant log forgery)". This is the API key's whole tenant boundary: it pins the caller 1:1 to exactly one `LogSource`, with no team/session concept at all. It gates both `POST /ingest/:sourceId` and `POST /ingest/:sourceId/raw` (router.ts:320-353).
- **Global admin.** `requireAdmin` (router.ts:1464-1477) resolves the userId then checks `prisma.user.findUnique(...).role === "ADMIN"` — a Postgres `UserRole`, unrelated to team membership. Gates every `/admin/*` route.

For team-scoped by-id write routes (state-changing, with a `:id`-shaped path param) the convention is: resolve the target resource's owning team, then call `requireTeamRole`, before mutating — not just "the caller is authenticated". `requireResourceTeam(loadTeamId, notFoundMessage)` (router.ts:1022-1035) is the shared factory: given a `loadTeamId(id): Promise<string | null>` resolver, it does load → 404 if the resource doesn't exist → `requireTeamRole` (403 if not a member) → return the teamId. Four instantiations exist: `requireRuleTeam`, `requireMaintenanceWindowTeam`, `requireIncidentTeam`, `requireIssueTeam` (router.ts:763, 871, 1083, 1327).

The factory alone does not stop a future route from skipping the check — nothing forces a handler to call it (router.ts's own comment at the factory site says exactly this). The actual, structural enforcement is `backend/tests/unit/router-team-scoping.test.ts`: it parses `router.ts` via the TypeScript compiler API (it never *imports* router.ts — importing would construct live Prisma/ClickHouse/Redis-backed services as a module-load side effect), extracts every `apiRouter.<method>("path", ...)` call, filters to state-changing (POST/PUT/PATCH/DELETE) routes with a `:param`, and requires each to appear in `ROUTE_TEAM_GUARDS` as either `{ kind: "resolver", resolverName }` (the route must contain a real AST `CallExpression` to that resolver — a mention in a comment does not count, pinned by regression tests) or a justified `{ kind: "allowlist", reason, verify? }`. Two things make this genuinely structural rather than a written convention: a route present in `router.ts` but absent from `ROUTE_TEAM_GUARDS` fails CI immediately, and an `apiRouter.*` registration shape the walker doesn't recognise (`.route()` chaining, `.use()` sub-router mounts, a non-string-literal path) also fails CI instead of silently escaping classification. Shipped as commit `1534e38` (PR #105); documented in German in `ENGINEERING.md:9-43` — this doc is the authoritative English write-up of the same convention.

## Where it's enforced

- **The precedent.** Four cross-tenant IDORs of this exact shape were found and fixed on 2026-07-12: PR #97 (invites), PR #98 (incident actions/timeline), PR #100 (alert-rule mute/unmute), PR #101 (maintenance-window delete). Three of the four had each hand-rolled a near-identical "load by id, check team" helper before PR #105 consolidated them into `requireResourceTeam`; PR #97 (invites) used inline per-route checks instead and still does (see below). A same-day companion fix, PR #102, closed the analogous *read*-path gap on `GET /query/jobs/:id` (router.ts:542-552: "requireAuth alone let any authenticated user read another team's query results (IDOR)") — a read path has no `:id`-write-route shape, so it sits outside this guard's coverage and is tracked separately (agent-tasks `9d86d755`).
- **Allowlist categories, with their justification.** `POST/PUT/PATCH/DELETE /ingest/:sourceId(/raw)` — no team/session exists at all; `authenticateApiKey` itself is the tenant boundary (verified via an `identifier` check that the middleware is actually wired, not a call). `PUT/POST/DELETE /logs/views/:id` and `/subscriptions/:id` — scoped per-(id, teamId) or per-(id, userId) at the Prisma-query level inside the service, so a foreign id paired with the caller's own scope matches nothing rather than needing a pre-check. `POST /teams/:id/invites` — the `:id` path param IS the team itself, not a separate resource to resolve; `requireTeamRole` runs directly against it, further narrowed by `canManageInvites` (below). `POST /invites/:token/accept` — authorization is the unforgeable, single-use, expiring invite token itself, a capability-token pattern, not a membership check (the accepting user isn't a member yet). `DELETE /invites/:id` — loads the invite inline and derives its team (the same resource-derived shape `requireResourceTeam` models), kept inline rather than forced onto the shared factory because it also needs `canManageInvites` on top of plain membership. `/admin/*` — `requireAdmin`'s global role gate supersedes team scoping by design.
- **Invite management needs more than membership.** `canManageInvites(role)` (router.ts:1372-1374) returns `role === "OWNER" || role === "ADMIN"`, gating `POST/GET /teams/:id/invites` and `DELETE /invites/:id` on top of `requireTeamRole`: an invite can grant membership up to role ADMIN and the list payload carries the join token, so mere membership must not be enough (the router's own comment names this as cross-tenant IDOR / privilege escalation prevention).

## Open, evolving — not decided here

`requireTeamRole` (router.ts:985-996, via `resolveTeamRole`, router.ts:968-979) checks **membership only** — it looks up `prisma.teamMember.findUnique` and returns whatever `TeamRole` (`OWNER | ADMIN | MEMBER | VIEWER`, `schema.prisma`) the row carries, without asserting a minimum. So, outside `canManageInvites` (OWNER/ADMIN only), shared saved-view management (`canManageShared`, OWNER/ADMIN, log-view-service.ts:160-162 — a VIEWER or MEMBER cannot mutate another user's shared view), and the admin routes (`requireAdmin`, a separate global gate), **a VIEWER currently passes the exact same write gates as MEMBER/ADMIN/OWNER everywhere `requireResourceTeam`/`requireTeamRole` is the only check** — mute/unmute an alert rule, acknowledge/resolve/reopen an incident, delete a maintenance window, reassign an issue, and so on. Whether VIEWER should be read-only, and where the line should sit, is an open, evolving question — tracked as agent-tasks `2a52b2b1`, operator decision pending. This doc states the fact without deciding it.
