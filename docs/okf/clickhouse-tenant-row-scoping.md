## The invariant

The four aggregation/search methods share `LogRepository.buildScopedWhere` (`backend/src/repositories/log-repository.ts:458-498`), which builds the `WHERE` clause as parameterised ClickHouse SQL. The **first** condition it pushes, unconditionally, is `team_id = {teamId:String}` (line 467) ŌĆö every other condition (`source_id`, `timestamp` range, ad-hoc filters, free-text search) is appended after it. The four query methods route through it: `search` (line 79-81), `getFacets` (145-154), `getHistogram` (177-178), and `getPatterns` (203-204) each call `this.buildScopedWhere(query)` before composing their own aggregation SQL around the resulting `where`/`params`. This is a second, independent tenancy layer *below* `router.ts`'s session/team authz (see [api-authz-and-team-scoping.md](api-authz-and-team-scoping.md)): even if a route-level check were ever missed, none of these methods can execute without a `team_id` predicate, because `teamId` is a required field of every query shape `buildScopedWhere` accepts and it is always conditions[0]. Three further public methods read the `logs` table WITHOUT `buildScopedWhere` and instead place the same `team_id = {teamId:String}` predicate first by hand: `getStats` (log-repository.ts:269, predicates at 278/283/288), `getContext` (:308, scope condition at :325), and `findById` (:572, predicate at :576). The real invariant is therefore "team_id is the first predicate in every logs read", not "every read goes through buildScopedWhere" ŌĆö an auditor must check all seven methods, not just the shared builder.

The clause is only as safe as the values that can reach it. Non-searchable filter fields go through `fields['<key>']`, ClickHouse's map-indexing syntax over the `logs` table's `Map(String, String)` column (`backend/clickhouse/init.sql`). Building that literal from a caller-controlled `filter.field` string is a WHERE-clause breakout risk if the key isn't sanitised ŌĆö a crafted key like `x'] OR 1=1 OR fields['x` could otherwise neutralise the parameterised `team_id` scope entirely. `buildFilterCondition` (log-repository.ts:400-452) closes this: any field not in `SEARCHABLE_COLUMNS` (`level`, `service`, `host`, `message`) is lowercased and stripped to `[^a-z0-9_-]` (log-repository.ts:428, "the same allow-list used for free-text search"); if nothing safe remains the filter is dropped (`return null`) rather than emitted with an empty key.

## Where it's enforced

- **`buildScopedWhere`** (log-repository.ts:458-498): pushes `team_id` first, then conditionally `source_id`, `timestamp >= / <=` (converted via `toChTimestamp`), each `filters[]` entry via `buildFilterCondition`, and ŌĆö only when `query.queryType === "sql"` ŌĆö a free-text search condition. All values are bound through ClickHouse's `{name:Type}` parameter syntax, never string-interpolated directly into the clause.
- **The map-key sanitiser** (log-repository.ts:422-432): the *same* regex allow-list (`[^a-z0-9_-]` stripped, lowercased) gates both the dynamic filter-field path and (per its own comment) the free-text search path, so there is one sanitisation rule to keep in sync, not two independently-maintained ones. This closed a real finding: PR #75's CHANGELOG entry documents "ClickHouse SQLi: `log-repository.ts buildFilterCondition` now sanitises the dynamic map key against the existing allow-list regex so a crafted `filter.field` cannot break out of the `fields['<key>']` literal" as one of four HIGH audit findings closed in that release.
- **`queryType: "sql"` is a legacy misnomer, not a raw-SQL surface.** `searchSchema` (`backend/src/validation/schemas.ts:36-49`) defines `queryType: z.enum(["sql", "natural"]).default("sql")`. Despite the name, the `"sql"` branch never accepts or executes caller-supplied SQL: `buildSearchCondition` (log-repository.ts:505-531) tokenizes the query string on whitespace, sanitises each token to `[a-z0-9_-]` (dropping tokens of length <= 1), expands a small set of stemming variants (`fail`/`failed`/`failure`/`failures`/`failing`), and ANDs together per-token ILIKE conditions over `message`/`service`/`host`, each bound as a parameter. "sql" here means "the structured/manual query mode" as opposed to `"natural"` (the NLQ-translated mode, see [nlq-pipeline.md](nlq-pipeline.md)) ŌĆö there is no code path where a user-supplied string is concatenated into the query as SQL.

## What breaks it

- A new `LogRepository` query method that composes its own `WHERE` clause without calling `buildScopedWhere` (or that calls it but then discards/reorders the returned string) would read across all teams rows with no tenant boundary at all ŌĆö there is no second gate to catch this. The same applies to edits of the three hand-scoped methods (findById/getContext/getStats): dropping their inline `team_id` predicate is equally fatal, and nothing shared would catch it. The tenancy contract is "team_id first in every logs read", enforced per-method.
- A new dynamic-field code path (e.g. a facet or sort field built from a caller string) that does not route through the same `[^a-z0-9_-]` allow-list sanitiser as `buildFilterCondition`/`buildSearchCondition` re-opens the WHERE-breakout class PR #75 closed. Two independently-drifting sanitisers (rather than the one shared regex) is itself a smaller version of the same risk.
- Treating `queryType: "sql"` as license to accept literal SQL from a request body ŌĆö it is a naming artifact from the tokenized free-text path, not a raw-SQL feature; adding real SQL passthrough here would be a new, unreviewed injection surface.
